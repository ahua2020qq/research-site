深度解读报告

一个人 + 70 个 AI Agent
2 天搭建完整银行系统

基于 GLM-4.7 模型的一次真实多 Agent 并行开发实测，70 个 Agent 在约 20 小时内完成了包含 8 个微服务的银行全栈系统。 本报告对原始运行数据进行深度解读，分析编排策略、Token 经济学、并行效率与工程启示。

关键发现

输入 Token 是输出的 ~7.5 倍——这些 Agent 绝大部分时间在阅读和理解现有代码/文档，而非生成新代码。这是代码生成类 Agent 的典型特征：理解成本远高于生成成本。

Explore 类 Agent 的输入占比最高（93%）——它们几乎不写代码，只读和分析。如果按 Token 单价计费，探索阶段是「纯成本」阶段，但它为后续生成阶段提供了关键的领域知识。

一个重要的成本优化方向：用轻量模型做探索，用强力模型做生成，可以显著降低总成本。

工程启示

测试修复循环不是 Bug，而是 Feature。

传统开发中，人工修 Bug 的成本往往被低估（隐性时间）。AI Agent 的优势在于：修复过程完全可量化——你能精确知道修了哪些 Bug、花了多少 Token、用了多久。

33% 的修复占比提示我们：在规划 AI Agent 项目时，至少要预留 1/3 的 Token 预算给测试和修复。

这种微服务拆分模式天然适合多 Agent 并行——每个服务边界清晰，Agent 之间几乎没有文件冲突。如果换成单体架构，70 个 Agent 同时改同一个代码库，冲突率会大幅上升，并行效率将大打折扣。

换句话说：微服务架构是 AI Agent 高效并行的必要前提之一。

Section 07

关键结论与工程启示

结论一
业务逻辑是并行收益最大的环节
各业务领域天然解耦，13 个 Agent 同时开发 8 个服务，几乎零冲突。未来规划时应优先在业务层使用多 Agent。
结论二
测试修复是 Token 消耗黑洞
33% 的 Agent 用于修复，且平均 Token 消耗比开发 Agent 高 32%。提升首次生成质量可直接降低总成本。
结论三
输入远大于输出——理解比生成贵
~88% 的 Token 用于输入（阅读代码/文档）。用轻量模型做阅读、强力模型做生成，是成本优化方向。
结论四
微服务架构是 Agent 并行的催化剂
清晰的模块边界让 Agent 不冲突。单体架构下多 Agent 并行的冲突率会显著拉低效率。
终极启示

这次实测证明了一个关键论点：AI Agent 不是替代程序员，而是提供了一种全新的工程组织形式。

传统上，组织 70 个开发者需要项目管理、代码审查、沟通协调等巨大的组织成本。而 70 个 Agent 的协调成本几乎为零——只要编排策略正确、模块边界清晰，它们就能高效协作。

未来的 AI 工程师，核心竞争力不再是写代码的速度，而是设计 Agent 编排策略、定义模块边界、优化 Token 预算的能力。
