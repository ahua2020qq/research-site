SDD 核心理念
权力倒置 · 规范六原则 · 三大层次
⚡ 权力倒置：最核心的思维转变
维度传统范式SDD 范式
权力关系规范服务于代码代码服务于规范
PRD 角色指导实现的参考文件直接生成实现的源头
调试方式修复代码修复产生了错误代码的规范
维护软件重构代码演进规范
事实来源代码是唯一事实规范是第一性产物
📌 SDD 六大核心原则
① 规范是通用语言
规范是首要制品，代码只是规范在特定语言和框架中的"表达"，不同语言产生不同实现，但规范唯一
② 可执行规范
规范必须足够精确、完整且无歧义，以至于能直接驱动可工作系统的生成，而非仅作参考
③ 持续精化
一致性验证不是一次性门禁，而是 AI 持续分析规范中歧义、矛盾与遗漏的过程
④ 研究驱动上下文
研究代理全程收集技术选型、性能影响和组织约束等关键上下文，在规范生成前完成
⑤ 双向反馈
生产环境的指标与故障不只触发热修复，而是驱动规范的持续演进，形成闭环
⑥ 分支探索
从同一规范可生成多个实现方案，探索性能、可维护性、用户体验等不同优化路径
📊 规范的三大层次（由浅入深）
Level 1
规范优先
Spec-First
先写规范再让AI生成代码；任务完成后规范可不保留
适合 小团队 · 快速迭代 · 探索期
工具 Kiro · OpenSpec 极简模式
Level 2
规范锚定
Spec-Anchored
规范在任务完成后持久保留，用于后续功能的演进和维护
适合 中型团队 · 持续演进系统
工具 OpenSpec · Spec-kit 标准模式
Level 3
规范即源码
Spec-as-Source
规范成为主要源文件，开发者只编辑规范，永远不直接修改代码
适合 核心基础设施 · 高可靠性系统
工具 Tessl · Spec-kit 激进模式

# 003 - 实时聊天功能
## 概述 (Overview)
实现聊天室实时消息推送，支持消息历史与用户在线状态。
## 用户故事 1 — 发送消息 (P1)
验收场景：
  • 假设 已登录用户在聊天室
  • 当 输入消息并按下回车
  • 则 消息在 500ms 内出现在窗口
## 边界条件
  • 消息超过 1000 字符时？
  • 网络断开时系统如何应对？

类比技术设计文档，定义系统"怎么做"。包含技术选型与理由、架构决策、数据模型、接口契约、性能要求等。

阶段一
第 1-4 周

🌱 试点验证 — 最小风险启动

以 1-2 名开发者 + 1 个非关键新特性为试点。选择低摩擦工具，验证团队可行性。

  • 选定 1 名 SDD 种子选手，完成工具安装与基础培训
  • 选择一个中等复杂度的新功能，完整跑通 spec/plan/tasks/implement
  • 使用 GitHub Copilot 或 Cursor（低学习曲线）降低摩擦
  • 记录时间消耗、质量指标、遇到的障碍
  • 复盘：提炼模板，识别团队培训需求
成功标准 完成特性 · 可衡量的时间节省 · spec/plan/tasks 三件套完整
阶段二
第 5-12 周

🚀 团队扩展 — 模式标准化

把验证过的模式扩展到全团队。建立团队级规范模板和评审流程，开展培训工作坊。

  • 建立团队 spec/plan/tasks 标准模板库（纳入 Git 仓库）
  • 制定规范评审流程（谁审 spec？谁审 plan？）
  • 开展规范写作工作坊（重点：如何写好验收标准）
  • 所有新功能强制走 SDD 流程，存量系统选择性引入
  • 建立 CONSTITUTION.md（项目架构宪法）
成功标准 50%+ 新特性采用 SDD · 规范评审流程运转正常
阶段三
第 13-24 周

⚙️ 全组织推广 — 成为默认工作流

将 SDD 设为团队默认工作方式。制定治理政策，融入 CI/CD 流程，建立持续度量机制。

  • 将规范评审纳入 PR 门禁（无 spec.md 不得合并）
  • CI/CD 集成：自动检测规范与代码的一致性（漂移检测）
  • 建立 SDD 实践知识库，沉淀踩坑经验
  • 定期 SDD 回顾会议，持续优化模板和流程
  • 建立 ROI 度量面板（时间、质量、一致性三维度）
成功标准 80%+ 采用率 · ROI 为正 · 规范库成为团队核心资产

⑥

⑦
示例 — 输入/输出示例、样例数据、使用场景

🛡️
五支柱验证框架
确保 AI 生成代码达到生产级标准的核心保障体系
🔐 支柱一 · 安全验证
  • SAST（静态应用安全测试）集成到 CI
  • 依赖漏洞自动扫描（npm audit / pip audit）
  • 硬编码密钥检测（如 git-secrets）
  • 输入校验与清洗，防 SQL 注入 / XSS
🧪 支柱二 · 测试要求
  • 强制最小单元测试覆盖率（建议 ≥ 80%）
  • API 集成测试 + 端到端测试
  • 边界场景覆盖（spec.md 中定义的所有边界）
  • 负载与性能测试
📏 支柱三 · 代码质量标准
  • 强制 lint/format（ESLint / Black / Ruff）
  • 圈复杂度测量，设置阈值告警
  • 文档完整性 + 命名规范检查
  • 架构模式一致性（对齐 CONSTITUTION.md）
⚡ 支柱四 · 性能验证
  • 响应时间目标度量（spec 中已定义 SLA）
  • 内存 / CPU 资源限制监控
  • 数据库查询优化检查（慢查询告警）
  • 缓存策略验证 + 负载测试基准
🚀 支柱五 · 上线就绪
  • 环境变量管理（零硬编码配置）
  • 结构化日志 + 链路追踪（可观测性）
  • 优雅降级 + 回滚方案
  • 监控告警配置验证
⚖️
评审准则 对 AI 生成代码与人类编写代码应用完全相同的质量标准。代码评审时，优先检查是否遵循了 spec.md 中的规范要求。

避坑指南 · 常见误区
工程团队在落地 SDD 时最容易踩到的坑
📄
把 README 当规范
README 是给人看的散文，AI 无法从中稳定解析出可执行信息
→ 使用结构化 DSL 或 Markdown Schema，必须有验收标准
📝
规范写成散文叙述
"支持用户登录" 这类叙述性表达缺乏可验证性，会导致AI自由发挥
→ 统一使用 GIVEN/WHEN/THEN 格式描述行为，配合示例数据
🔄
规范与代码不做漂移检测
迭代后代码悄悄偏离规范，规范成为"历史文档"而非事实来源
→ CI 集成漂移检测工具，每次 PR 自动对比规范与实现一致性
📦
规范没有版本化策略
规范文件未纳入 Git 管理，无法回溯历史、无法 Diff 变更
→ 规范文件与代码在同一仓库同一 PR 中提交，联动版本化
🔍
过度细化规范
把每行代码的实现细节都写进规范，维护成本极高，团队反感
→ 规范描述"做什么"和"为什么"，不描述"怎么做"（那是 plan.md 的职责）
🚀
一开始就全面推广
跳过试点直接全团队推行，遇到阻力时容易全面失败
→ 严格按"试点→扩展→推广"三阶段推进，先用1-2人小范围验证
🤖
过度依赖AI生成的中间产物
AI生成的 spec.md 直接进入 plan 阶段，没有人工审核校验
→ 规范必须由团队成员审核确认后才能进入下一阶段，AI是助手不是决策者
🏛️
没有建立项目宪法
缺少 CONSTITUTION.md，每个人对架构原则理解不一，规范质量参差不齐
→ 在第一个特性开始前，先建立 CONSTITUTION.md，定义核心约束与原则
🎯 SDD 适用场景快速判断
✅ 强烈适合 SDD
  • 多人协作的功能开发
  • 生命周期长的核心业务系统
  • 稳定性要求高（金融、医疗、基础设施）
  • AI 编码纳入主开发流程
  • 需要频繁审计和合规的项目
❌ 不建议强制用 SDD
  • 快速原型验证（探索性开发）
  • 一次性脚本或临时工具
  • 3 行以内的 Bug Fix
  • 纯个人项目且不涉及协作
  • 极短生命周期的演示 Demo

🔗 SDD 与其他开发方法论的关系
🧩
SDD 不是替代 TDD / BDD / DDD，而是在更高层次上整合它们，共同解决不同维度的"对齐"问题。
方法论核心保证关注层次与 SDD 的关系
TDD 代码是对的 单元 / 函数级 SDD 的 tasks.md 强制 [T] 先于 [I]，天然包含 TDD
BDD 做的事情是对的 业务行为级 spec.md 的 GIVEN/WHEN/THEN 格式正是 BDD 验收标准
DDD 领域概念是对的 领域模型级 plan.md 和 data-model.md 承载领域模型的设计决策
SDD 整个系统，从意图到实现，始终对齐 全链路级 规范是唯一事实来源，统一所有层次的对齐
先对齐，再构建
Agree Before You Build — 在 AI 能将高质量规范直接转化为可工作代码的今天，一份清晰精确的规范，是从"随兴编码"迈向"可预测工程"的关键一步。

🔒 代码服务于规范
🔄 规范随系统持续演进
